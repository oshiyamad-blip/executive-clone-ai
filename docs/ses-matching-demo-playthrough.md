# SES案件・要員マッチングシステム デモプレイスルー記録

作成日: 2026-07-17 ／ プレイヤー: Sonnet（1日2回しかメールを見られない少人数運用のSES営業担当を想定）

本書は `npm run ses:demo` を実際に実行し、コンソール出力・生成物（JSON・下書きメール）を
ユーザ目線で観察した記録である。捏造なし、実際に出た出力のみを引用する。

---

## 1. シナリオ

私は小さなSES事業者の営業担当。日中は客先常駐や商談で外に出ていて、メールをチェックできるのは
朝と夕方の1日2回だけ。その間に他社の営業から案件情報・要員情報のメールが何通も届く。
これまでは1通ずつ読んで、Excelに手で転記し、単金と勤務地を突き合わせて「これいけるかも」と
思ったペアを見繕って紹介メールを書いていた。この機能が、その突合と下書き作成を自動でやって
くれるというので、まずは実データ無しの `ses:demo` で挙動を確認する。

---

## 2. 実行①: 既定設定（粗利下限10万円）でそのまま実行

```
$ npm run ses:demo

> executive-clone-ai@0.1.0 ses:demo
> DEMO_MODE=true tsx src/ses/index.ts

=== SES案件・要員マッチングバッチ開始 ===
モード: DEMO（外部呼び出しなし） / 粗利下限: 100000円/月 / 候補上限: 5件
SES収集: 未処理メール7件

=== SESマッチング結果サマリ ===
検出日時: 2026/7/17 15:30:38
成立候補: 1件 / 要確認: 1件

【成立候補】
・大手ECサイト バックエンド刷新案件 × K.S. — 粗利15.0万円/月, 適合スコア100点
  根拠: スキル一致率100%・勤務地適合・時期適合に基づく機械判定です。
  案件側下書き: /home/user/executive-clone-ai/data/ses-demo/drafts/demo_draft_1.txt
  要員側下書き: /home/user/executive-clone-ai/data/ses-demo/drafts/demo_draft_2.txt

【要確認（単金・勤務地等が不明のため人による確認が必要）】
・大手ECサイト バックエンド刷新案件 × N.Y. — 単金または勤務地情報が不足しているため要確認です（スキル一致率100%）。


=== SESバッチ完了: マッチ候補 計2件 ===
```

起動直後に「モード: DEMO（外部呼び出しなし）」「粗利下限」「候補上限」が出るのがありがたい。
今どの設定で動いているか一目でわかるし、これがdemoなのか本番なのかを見間違えない。

「未処理メール7件」とあるので、fixtureとして案件・要員あわせて7通ぶんのメールが投入されている
らしい。実際の運用でも、朝チェックしたときに「昨夜からの新着7件」というのが最初に出てくる
イメージだと思う。

### マッチ結果と下書きメールの実物

`data/ses-demo/matches.json` の成立ペア:

```json
{
  "id": "match_proj_demo_p1_eng_demo_e1",
  "title": "大手ECサイト バックエンド刷新案件 × K.S.",
  "grossMarginJpy": 150000,
  "score": 100,
  "reason": "スキル一致率100%・勤務地適合・時期適合に基づく機械判定です。",
  "needsReview": false,
  ...
}
```

粗利15万円（案件単金上限75万 − 要員希望単金60万 = 15万）が下限10万円を超えているので成立。
案件側営業への下書き `demo_draft_1.txt` の実物:

```
To: tanaka@alphatech.example.jp

田中一郎様

いつもお世話になっております。
貴社ご案内の案件「大手ECサイト バックエンド刷新案件」につきまして、以下要員をご提案いたします。

■ご提案要員
表示名: K.S.

経験年数: 6年

■マッチ判定

```

これはそのまま送れそうなレベルの文面。「★送付前に単金開示の要否をご確認ください」という
一言が入っているのも実務的で助かる（単金をそのまま先方に見せていいかは案件・商流によって
違うので、これを機械的に送信せず一度人間が見るという設計は正しいと感じた）。

要員側営業への下書き `demo_draft_2.txt` も同様に、案件概要（必須/尚可スキル・単金レンジ・
勤務地・開始時期・商流メモ）を要約した文面になっていた。

「要確認」枠（N.Y.さん）の方は下書きが作られておらず、サマリに理由（単金不明）だけが出る。
これは自分で単金を確認してから手動で紹介するかどうかを判断しろ、ということだと理解した。
LLMコールを節約する設計だと設計書にも書いてあったので、意図通りの挙動。

## 3. 実行②: 粗利下限を変えて「10万円は変更できる」を検証

デフォルトの10万円のときは1件成立（K.S.案件、粗利15万円）だった。これを実際に変えて
挙動が変わるか確認した。

### 5万円に下げる

```
$ MIN_GROSS_MARGIN_JPY=50000 npm run ses:demo
...
モード: DEMO（外部呼び出しなし） / 粗利下限: 50000円/月 / 候補上限: 5件
...
成立候補: 1件 / 要確認: 1件
```

5万円にしても成立は1件のまま（K.S.案件のみ、粗利15万円）。他のペアは粗利2万円（大阪の
Java案件×M.T.さん）しかなく、5万円の下限も超えられなかったので変化なし。

### 20万円に上げる

```
$ MIN_GROSS_MARGIN_JPY=200000 npm run ses:demo
...
モード: DEMO（外部呼び出しなし） / 粗利下限: 200000円/月 / 候補上限: 5件
...
成立候補: 0件 / 要確認: 1件

=== SESバッチ完了: マッチ候補 計1件 ===
```

20万円まで上げると、さっきまで成立していた15万円のペアも足切りされ、成立0件になった。
「粗利下限を厳しくしたら成立が減る」が実際に体験できた。0件でもエラーにならず、ちゃんと
「成立候補: 0件」とサマリが出るのも安心できる（0件だと何も出ずに終わって気づかない、という
事故がなさそう）。

### 1.5万円まで下げてもう1ペア通す

境界を確かめたくて、さらに下限を1.5万円まで下げてみた。

```
$ MIN_GROSS_MARGIN_JPY=15000 npm run ses:demo
...
モード: DEMO（外部呼び出しなし） / 粗利下限: 15000円/月 / 候補上限: 5件
...
成立候補: 2件 / 要確認: 1件

【成立候補】
・大手ECサイト バックエンド刷新案件 × K.S. — 粗利15.0万円/月, 適合スコア100点
  ...
・金融系基幹システム保守案件 × M.T. — 粗利2.0万円/月, 適合スコア100点
  根拠: スキル一致率100%・勤務地適合・時期適合に基づく機械判定です。
  案件側下書き: .../demo_draft_3.txt
  要員側下書き: .../demo_draft_4.txt

=== SESバッチ完了: マッチ候補 計3件 ===
```

粗利2万円の大阪案件（金融系基幹システム保守案件×M.T.さん）が新たに成立候補として拾われ、
下書きも自動で作られた。10万→5万→20万→1.5万と一通り動かして、環境変数
`MIN_GROSS_MARGIN_JPY` を変えるだけでマッチ件数の閾値が実際に動くことを確認できた。
「10万円は初期値で、変更できる」という要求はデモ上でも実証されている。

## 4. おまけ: `--collect-only` / `--match-only` の部分実行も試した

1日2回しか見られない運用だと「朝は収集だけしておいて、夕方まとめて突合」といった分割
運用がしたくなるかもしれないと思い、試した。

```
$ DEMO_MODE=true npx tsx src/ses/index.ts --collect-only
=== SES案件・要員マッチングバッチ開始 ===
モード: DEMO（外部呼び出しなし） / 粗利下限: 100000円/月 / 候補上限: 5件
SES収集: 未処理メール7件
=== --collect-only指定のため収集・保存のみで終了（案件4件・要員3件） ===

$ ls data/ses-demo/
engineers.json  projects.json

$ DEMO_MODE=true npx tsx src/ses/index.ts --match-only
=== SES案件・要員マッチングバッチ開始 ===
モード: DEMO（外部呼び出しなし） / 粗利下限: 100000円/月 / 候補上限: 5件

=== SESマッチング結果サマリ ===
検出日時: 2026/7/17 15:31:27
成立候補: 1件 / 要確認: 1件
...
=== SESバッチ完了: マッチ候補 計2件 ===
```

`--collect-only` で保存されたJSONを `--match-only` がそのまま読み込んで突合できた。
これなら「朝は収集だけ流しておく」「夕方に粗利下限を変えて何度も突合し直す」といった
使い方ができそう。

## 5. 良かった点

- **モード表示が最初に出る**: 「DEMO（外部呼び出しなし）」と出るので、うっかり本番相当だと
  誤解することがない。粗利下限・候補上限も同時に出るので、今どの設定で動いているか一目瞭然。
- **0件でも黙らない**: 粗利下限を厳しくして成立0件になっても、ちゃんと「成立候補: 0件」と
  出た。何も表示されず終わる、という事故が起きにくい。
- **下書きがそのまま送れそうなレベル**: 案件側・要員側それぞれの立場に合わせた文面になって
  おり、要員の単金を先方にそのまま見せていいか確認を促す一文が入っているのも実務的。
- **要員の匿名化（イニシャル表示）**: 個人情報への配慮が感じられた。
- **粗利下限が本当に環境変数だけで変わる**: 5万・10万・20万・1.5万と試して、閾値をまたぐ
  ところで実際にマッチ件数が変わることを自分の目で確認できた。「あとで基準を変えたい」
  というニーズに応えられている。
- **部分実行（`--collect-only`/`--match-only`）が地味に便利**: 1日2回しかメールを見ない
  運用だと、収集と突合を別タイミングでやりたくなるはずなので、この分割ができるのは実務に
  合っている。

## 6. 分かりにくかった点・気になった点

- **「要確認」枠に下書きが作られない理由が初見では分からない**: サマリの文言（「単金または
  勤務地情報が不足しているため要確認です」）を読めば理由は分かるが、初めて使うときは
  「なぜこのペアだけ下書きが無いのか」と一瞬迷うかもしれない。ヘルプやREADMEに
  「要確認枠はLLM判定・下書き作成をスキップする設計」だと一言書いてあると安心できそう。
- **「粗利15.0万円/月」の内訳が見えない**: サマリ上は粗利の最終額しか出ないので、「案件単金
  上限はいくらで、要員希望単金はいくらだったからこの粗利になったのか」を確認したいときは
  `projects.json`/`engineers.json` を別途開いて自分で引き算する必要があった。サマリ表示に
  内訳（例: 「75万−60万=15万」）が添えられているとより一目で納得できる。
- **「候補上限5件」がdemoデータでは体感できない**: 今回のfixtureでは1案件あたりの候補が
  最大でも2件程度だったため、候補上限（既定5件）で足切りされる場面は見られなかった。
  実データで案件・要員の件数が増えてきたときに、この上限が効いているかどうかは別途
  確認したい。
- **メール本数（7件）と構造化後の件数（案件4件+要員3件）の対応が一見で分かりにくい**:
  「未処理メール7件」→「案件4件・要員3件」という内訳がログに出るのは助かるが、
  1通のメールから複数案件が抽出されるケース（実際 `proj_demo_multi_1`/`proj_demo_multi_2`
  は同じ `sourceMailId: "sesmail_demo_multi"` から2件抽出されていた）があることは、
  JSONを開いて初めて気づいた。サマリ表示に「うち1通から2案件抽出」のような注記が
  あると分かりやすいかもしれない。

## 7. 実運用への要望

- 実際のメールで動かしたときに、抽出精度（スキル・単金・勤務地の読み取り漏れ）がどの程度
  出るのかを早めに検証したい。特に単金の「応相談」「スキル見合い」といった曖昧な表記の
  扱われ方（null化）が想定通りかは実データで見てみたい。
- 粗利下限や候補上限のような数値設定は、日々の営業感覚で微調整したくなりそうなので、
  `.env` 直接編集以外に、都度コマンドラインで一時的に上書きできる今のやり方（環境変数指定）
  で十分使いやすいと感じた。今回のように `MIN_GROSS_MARGIN_JPY=15000 npm run ses:demo`
  と打つだけで試せるのは良い。
- 「要確認」枠がどんどん溜まっていったときに、まとめて確認しやすい一覧表示（例えば
  「要確認だけをCSVでエクスポート」）があると、1日2回しか見られない身としては助かりそう。

---

## 8. まとめ

`npm run ses:demo` は外部API・認証情報が一切ない環境で、案件4件・要員3件の構造化、
成立1件・要確認1件（既定の粗利下限10万円時）の検出、成立ペアへの紹介メール下書き2通の
生成まで、エラーなく完走した。粗利下限を5万円・20万円・1.5万円に変えて実行し、
20万円では成立0件、1.5万円では成立2件（+要確認1件）に変化することを確認し、
「粗利下限10万円は変更可能」という要求が実際に機能していることを確かめられた。
`--collect-only` / `--match-only` の部分実行、下書きメールの実物（案件側・要員側それぞれ）も
実際に生成・確認済み。営業担当としては、Excel転記と粗利計算の手間が減り、下書きの叩き台が
自動で出てくるだけでもかなり助かる、という印象。
