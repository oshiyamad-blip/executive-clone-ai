# SES案件・要員マッチングシステム 要件定義・設計書

作成日: 2026-07-17 ／ ステータス: レビュー待ち（実装は次フェーズ）

## 1. デザイン思考による設計プロセス

本書は、思いつきの機能列挙ではなくデザイン思考の5段階（共感 → 問題定義 → アイデア出し → プロトタイプ → テスト）に沿って要件を導出した。

### 1.1 共感（Empathize）— 誰のどんな状況か

**ペルソナ: SES事業の営業担当（少人数運用）**

- Xserver 上のメールアドレスに、パートナー企業の営業から「案件情報」と「要員（エンジニア）情報」のメールが毎日大量に届く。将来的には 1日500通 規模を想定。
- 情報の形式はバラバラ: メール地の文、添付 Excel、添付 PDF、Google スプレッドシートのリンク。単金の表記も「〜80万」「スキル見合い」「時間単価4,500円」など揺れが大きい。
- メールを確認できるのは 1日2回程度。その間に届いた案件と要員を頭の中で突合しており、**粗利の取れる組み合わせを見逃している**確信がある。
- マッチを見つけた後も、双方の営業へ紹介メールを書く作業に時間を取られる。

### 1.2 問題定義（Define）— POV と How Might We

> **POV**: 少人数の SES 営業は、1日2回しかメールを見られないのに、非構造化された大量の案件・要員情報から粗利条件を満たすペアを人力で見つけなければならない。

- **HMW1**: どうすれば形式バラバラのメール（本文・Excel・PDF・スプシ）を、比較可能な構造化データに自動変換できるか？
- **HMW2**: どうすれば「案件単金 − 要員単金 ≥ 粗利下限（初期値10万円）」を満たすペアだけを、見逃しなく・ノイズ少なく提示できるか？
- **HMW3**: どうすればマッチ発見から紹介メール送付までのリードタイムを、1日2回の確認サイクル内に収められるか？
- **HMW4**: どうすればメール量が25倍（20通→500通/日）になっても運用コストが破綻しない設計にできるか？

### 1.3 アイデア出し（Ideate）— 方式の比較と採否

| 方式 | 内容 | 採否 | 理由 |
|---|---|---|---|
| A. 全文 LLM 突合 | 溜まったメール全文を LLM に渡し「マッチを探して」と依頼 | ❌ | 入力トークンがメール数の2乗オーダーで爆発。500通/日では成立しない。精度も不安定 |
| B. 構造化 → ルールマッチ → LLM 最終判定 | 各メールを1回だけ LLM で構造化し、突合はコード（無料）で行い、通過ペアのみ LLM で最終判定 | ✅ **採用** | LLM コールがメール数に対して線形。粗利条件は算術比較なので LLM 不要。API 費が最小 |
| C. 埋め込みベクトル検索 | メールをベクトル化し類似検索でマッチ | ❌ | 「粗利10万円以上」のような数値条件を扱えない。別途ベクトルDBの運用コストが増える |
| D. 人間がスプシに転記 → 関数で突合 | 手作業で構造化 | ❌ | 現状の課題（人力の限界）を解決しない |

### 1.4 プロトタイプ方針（Prototype）

- 既存リポジトリ executive-clone-ai の資産（Gmail 収集・Claude 構造化出力・Notion ラッパー・バッチ運用）を最大限再利用し、新規実装を最小化する（§6.3 再利用マップ）。
- Phase を3段階に分け、各 Phase 単体でも価値が出る構成にする（§9 ロードマップ）。
- テスト（Test）は Phase 1 完了時に実メール数日分で抽出精度を検証し、抽出スキーマとプロンプトを改善するサイクルを回す。

## 2. 背景と目的

- **ビジネスモデル**: 案件の単金と要員の単金の差額を粗利として確保する。判定基準は「案件単金 − 要員希望単金 ≥ **粗利下限**」。粗利下限の初期値は **10万円/月** とし、**設定で変更可能にする**（§6.5）。
- **目的**: メールで届く案件・要員情報の突合と紹介メール作成を自動化し、(1) 粗利の取れるマッチの見逃しをなくす、(2) 確認〜紹介のリードタイムを1営業日以内にする、(3) 運用コスト（API費）をメール量に対して線形かつ低額に保つ。

## 3. システム全体像

```
Xserverメール ──(自動転送)──▶ Gmail
                               │  ①収集 collect（1日2回 cron）
                               ▼
              本文 + 添付(Excel/PDF) + スプシリンク
                               │  ②展開 parse（xlsx→テキスト / PDF→documentブロック / Sheets API）
                               ▼
                     Claude Haiku 4.5（構造化出力）
                               │  ③抽出 extract（分類+抽出を1コールで）
                               ▼
        ┌─────── Notion ────────┐
        │  案件DB    要員DB      │   ④保存 store（名寄せ・重複排除込み）
        └───────┬───────────────┘
                │  ⑤突合 match（ルールベース: 粗利・スキル・居住地・時期 → 無料）
                ▼
        候補ペア ──▶ Claude Sonnet 5 で最終判定（少数のみ）
                │
                ▼
        ⑥生成 draft: 紹介メール2通（案件側宛・要員側宛）を Sonnet 5 で作成
                │        └▶ Gmail 下書きフォルダに保存（自動送信はしない）
                ▼
        ⑦通知 notify: マッチ結果DB（Notion）更新 + サマリメールを利用者へ送信
```

## 4. 機能要件

| ID | 機能 | 内容 | 優先度 |
|---|---|---|---|
| F1 | メール収集 | 対象アドレス宛のメールを1日2回（例: 8:00 / 17:00）取得。取得済み ID を管理し二重処理を防止 | 必須 |
| F2 | 添付・リンク展開 | 添付 Excel(.xlsx/.xls) はライブラリでテキスト化、PDF は Claude の document ブロックで直接読解、本文中の Google スプレッドシートリンクは Sheets API で読み取り | 必須 |
| F3 | 情報抽出・構造化 | メール1通につき LLM 1コールで「案件/要員/その他」の分類と項目抽出を同時に実施。構造化出力（JSON Schema）で受け取る。1通に複数案件・複数要員が含まれるケースに対応（配列で返す） | 必須 |
| F4 | マッチング | ルールベースで候補を絞り込み（粗利下限・スキル・居住地/リモート・稼働時期）、通過ペアのみ LLM で適合度を最終判定（スコア 0〜100 と根拠文） | 必須 |
| F5 | 紹介メール下書き | 成立マッチごとに、案件側営業宛（要員のご提案）と要員側営業宛（案件のご紹介）の2通を生成し、Gmail の下書きとして保存。**自動送信はしない** | 必須 |
| F6 | 結果通知 | マッチ結果を Notion マッチDBに記録し、バッチ実行後にマッチ一覧サマリを利用者宛にメール送信（0件でも実行結果を通知） | 必須 |
| F7 | 設定変更 | 粗利下限（初期値10万円）、バッチ時刻、候補ペア上限数などを `.env` で変更可能 | 必須 |
| F8 | 名寄せ | 同一案件・同一要員が複数の営業経路から届いた場合に重複統合（既存 `src/dedup/` の類似度判定を再利用） | 推奨 |
| F9 | ステータス管理 | 案件（募集中/終了）、要員（提案可/決定済）、マッチ（未確認/紹介済/成約/見送り）を Notion 上で手動更新でき、終了分は突合対象から除外 | 推奨 |

| スキル・経験年数 | string[] / number | |

### 4.3 マッチングルール（一次選抜・LLM 不使用）

1. **粗利条件**: 案件単金上限 − 要員希望単金 ≥ `MIN_GROSS_MARGIN_JPY`（初期値 100,000円）。単金不明のものは「要確認」として別枠提示
2. **スキル**: 案件の必須スキルと要員スキルの一致率が閾値以上（正規化後の突合）
3. **勤務地**: 案件がリモート可、または要員居住地が通勤圏（都道府県ベースの隣接判定）
4. **時期**: 案件開始時期と要員稼働可能日が整合
5. 1アイテムあたりの候補ペア数は上位 `MAX_CANDIDATES_PER_ITEM`（初期値5）件に制限 → LLM コスト上限を保証

## 5. 非機能要件

| 項目 | 要件 |
|---|---|
| コスト | 500通/日でも API 費が月3万円以内（§8 で試算）。LLM コール数はメール数に対して線形であること |
| 個人情報保護 | 要員情報は個人情報（PII）。Anthropic API は**ゼロデータリテンション（ZDR）契約のキーを推奨**（CLAUDE.md の方針と同じ。注: ZDR は Haiku/Sonnet で利用可）。Notion のアクセス権は最小限に |
| 縮退動作 | 環境変数未設定・外部 API 障害時は該当ステップをスキップして継続（既存流儀: `?? ''` / 早期 return / `Promise.allSettled`） |
| レート制限 | Notion 3req/s（既存ラッパーで対応済み）、Anthropic 429/529 は指数バックオフ（SDK 標準リトライ） |
| 誤処理耐性 | 紹介メールは必ず下書き止まり。マッチは人間の最終確認を経てから送信（§10） |
| 再実行安全性 | 処理済みメール ID の永続化により、バッチ再実行しても二重登録・二重下書きが発生しない |

## 6. アーキテクチャ設計

### 6.1 メール取得方式の比較（推奨: A案）

| | **A案: Xserver → Gmail 自動転送 + Gmail API（推奨）** | B案: Xserver IMAP 直接続 |
|---|---|---|
| 設定 | Xserver 管理画面で対象アドレスの自動転送を1回設定 | 不要 |
| 実装 | 既存 `src/collectors/googleAuth.ts` / `email.ts` を再利用。追加は添付取得（`messages.attachments.get`）のみ | node-imap 等を新規導入。MIME/添付パース・既読管理を自前実装 |
| 下書き作成 | 同じ Gmail API で下書き作成まで完結 | 下書き保存先が別途必要（結局 Gmail/SMTP が要る） |
| 検索・ラベル | Gmail のクエリ・ラベルで処理済み管理が容易 | UID 管理を自前実装 |
| リスク | 転送設定を止めると収集が止まる（監視で検知可能） | Xserver の IMAP 同時接続制限・仕様変更の影響 |

**A案を推奨する理由**: 実装量が最小（既存コードの拡張のみ）、収集から下書き作成まで単一の認証基盤（SA+DWD）で完結、運用時のトラブルシュートが Gmail UI 上で可能。

### 6.2 パイプラインとモジュール構成

既存の「機能ディレクトリ + `index.ts` + npm script」の流儀を踏襲する。

| ステップ | モジュール（新規/拡張） | 内容 |
|---|---|---|
| ① collect | `src/collectors/email.ts`（拡張） | 対象アドレスのメール取得 + 添付ダウンロード。`RawLog` に添付テキストを同梱 |
| ② parse | `src/ses/parse.ts`（新規） | xlsx→テキスト化（`xlsx` パッケージ）、スプシリンク検出→Sheets API 読取、PDF は base64 のまま次段へ |
| ③ extract | `src/ses/extract.ts`（新規） | Haiku 4.5 + 構造化出力で分類+抽出。既存 `src/llm/index.ts generateJson()` を利用（モデル指定引数を追加） |
| ④ store | `src/database/index.ts`（拡張） | 案件DB/要員DBへ保存。`saveSignal` と同型の `saveProject` / `saveEngineer` を増設。名寄せは `src/dedup/` 再利用 |
| ⑤ match | `src/ses/match.ts`（新規） | ルールベース一次選抜（無料）→ 候補ペアを Sonnet 5 で最終判定 |
| ⑥ draft | `src/ses/draft.ts`（新規） | 紹介メール2通生成 → Gmail 下書き作成 |
| ⑦ notify | `src/ses/notify.ts`（新規） | マッチDB更新 + サマリメール送信 |

npm scripts: `"ses:collect"`, `"ses:match"`, `"ses:daily"`（①〜⑦一括）。cron/systemd timer で1日2回実行（既存 `deploy/systemd/` の流儀）。

### 6.3 既存資産の再利用マップ

| 既存資産 | 再利用内容 |

### 6.4 Notion DB 設計

**案件DB**: 案件名(title) / 必須スキル・尚可スキル(multi_select) / 単金下限・上限(number) / 勤務地(rich_text) / リモート(select) / 開始時期(rich_text) / 商流メモ(rich_text) / 営業元会社・担当・メール(rich_text) / 元メールID(rich_text) / 受信日(date) / ステータス(select: 募集中・終了)

**要員DB**: 表示名(title, イニシャル推奨) / スキル(multi_select) / 経験年数(number) / 希望単金(number) / 居住地(rich_text) / 稼働開始可能日(date) / リモート希望(select) / 営業元(rich_text) / 元メールID(rich_text) / 受信日(date) / ステータス(select: 提案可・決定済)

**マッチ結果DB**: マッチ名(title: 「案件名 × 要員名」) / 案件(relation) / 要員(relation) / 粗利額(number) / 適合スコア(number 0-100) / 判定根拠(rich_text) / 下書きURL・下書きID(rich_text ×2通分) / ステータス(select: 未確認・紹介済・成約・見送り) / 検出日時(date)

### 6.5 可変設定（`.env` — 既存流儀のインライン env 方式）

```bash
# ===== SESマッチング設定 =====
MIN_GROSS_MARGIN_JPY=100000        # 粗利下限（円/月）← 要望の「10万円、変更可能に」
SES_TARGET_GMAIL=ses@example.com   # 転送先Gmailアドレス
SES_NOTIFY_TO=you@example.com      # サマリ通知の宛先
MAX_CANDIDATES_PER_ITEM=5          # 1アイテムあたりLLM判定に回す候補ペア上限
SKILL_MATCH_THRESHOLD=0.6          # 必須スキル一致率の下限
HOURLY_TO_MONTHLY_HOURS=160        # 時給→月額換算の稼働時間
NOTION_PROJECT_DB_ID=...
NOTION_ENGINEER_DB_ID=...
NOTION_MATCH_DB_ID=...
ANTHROPIC_MODEL_EXTRACT=claude-haiku-4-5   # 抽出用（切替可能）
ANTHROPIC_MODEL_MATCH=claude-sonnet-5      # 最終判定・メール生成用
USE_BATCH_API=false                # trueでBatch API（50%割引）を使用
```

設定値が増えるため、読み出しは `src/ses/config.ts` に一元化する（`Number(process.env.MIN_GROSS_MARGIN_JPY ?? '100000')` 形式）。

## 7. モデル選定と API 節約設計

**節約の基本方針**: 「LLM を呼ばないのが最大の節約」。突合の大部分（粗利の算術比較・スキル集合の一致・都道府県判定）はコードで無料実行し、LLM は (a) 非構造→構造の変換、(b) 少数候補の最終判定、(c) 対外品質が必要な文章生成、の3点にだけ使う。

### 7.1 用途別モデル（2026-07 時点の公式単価）

| 用途 | モデル | 単価（入力/出力 per MTok） | 選定理由 |
|---|---|---|---|
| 分類+抽出（全メール） | **Claude Haiku 4.5** | $1 / $5 | 呼び出し回数が最多の箇所。定型的な項目抽出は Haiku で十分な精度。構造化出力（JSON Schema）でパース信頼性を担保 |
| マッチ一次選抜 | **LLM 不使用** | ¥0 | ルールベース（§4.3）。ここを LLM にしないことが最大のコスト削減 |
| マッチ最終判定 | **Claude Sonnet 5** | $3 / $15（2026-08-31まで導入価格 $2 / $10） | 件数が少なく（候補ペアのみ）、スキルの文脈適合など判断品質が成約率に直結 |
| 紹介メール生成 | **Claude Sonnet 5** | 同上 | 対外文書のため品質重視。件数は成立マッチ×2通のみ |

Opus 4.8（$5/$25）は本用途ではオーバースペックのため不採用。抽出精度が不足した場合のみ `ANTHROPIC_MODEL_EXTRACT` を Sonnet に切り替えて比較検証する（設定で切替可能にしておく）。

### 7.2 コール数を減らす設計

1. **分類と抽出を1コールに統合**: 「このメールは案件/要員/その他か」+「該当項目の抽出」を単一スキーマで返す → コール数半減
2. **1メール複数件対応**: スキーマを配列で定義し、1通に案件が5件並記されていても1コールで処理
3. **候補ペア上限**: `MAX_CANDIDATES_PER_ITEM` で LLM 判定に回る件数に上限を設け、コストの暴走を構造的に防止
4. **Batch API（50%割引）**: 1日2回のバッチ処理でリアルタイム性が不要なため、Message Batches API が適用可能（ほとんどのバッチは1時間以内に完了、最大24時間）。§8 の試算に半額列を併記。Phase 1 は実装が単純な同期呼び出しで開始し、メール量が増えた段階で移行するのが現実的
5. **プロンプトキャッシュは限定的**: 抽出システムプロンプトは1〜2Kトークン程度で、Haiku 4.5 の最小キャッシュ対象（4096トークン）に届かないため効果は薄い。スキル正規化辞書等でプロンプトが4Kを超えたら再検討

### 7.3 添付・リンクのコスト特性

| 形式 | 処理方法 | LLM コスト影響 |
|---|---|---|
| Excel (.xlsx/.xls) | `xlsx` パッケージでテキスト化（無料） | テキスト分のみ。表が大きい場合は関連シートのみ抽出 |
| PDF | Claude の document ブロックで直接読解 | **ページ数比例で増加**（1ページ ≈ 1,500〜3,000トークン、画像+テキストで課金）。スキルシートは通常1〜3ページで許容範囲 |
| スプシリンク | Sheets API（無料）でテキスト化 | テキスト分のみ |

## 8. ランニングコスト試算

**前提**（試算式を明記するので、実測後に係数を差し替えて再計算できる）:

- 為替: 150円/USD、月30日稼働
- 抽出（Haiku 4.5）: 全受信メールに1コール。平均入力3,000トークン（システムプロンプト+本文+添付テキスト）、出力500トークン
  → 1通 = 3,000×$1/M + 500×$5/M = **$0.0055（約0.83円）**
- 最終判定（Sonnet 5 標準価格）: 候補ペア数 = 受信数×0.3/日 と仮定。入力1,500・出力300トークン
  → 1ペア = **$0.009（約1.35円）**
- 紹介メール生成（Sonnet 5）: 成立マッチ = 受信数×0.1/日 と仮定、1マッチ2通。1通あたり入力2,000・出力800トークン
  → 1マッチ = $0.018×2 = **$0.036（約5.4円）**

**感度・注意点**:

- 500通/日シナリオは「成立マッチ50件/日」を仮定した**上限見積り**。実際には提案可能な要員数で頭打ちになるため、判定・生成コストはこれより下振れする可能性が高い
- PDF 添付が多い場合（平均入力が3K→6Kトークン）、抽出コストは約1.6倍。逆に本文のみのメールが多ければ下振れ
- Sonnet 5 の導入価格（$2/$10、2026-08-31まで）適用中は判定・生成が約33%安い
- 非機能要件「500通/日で月3万円以内」は標準価格・同期呼び出しでも達成、Batch API 併用なら約1.3万円

## 10. 運用上の注意

1. **下書きは自動送信しない**: 紹介メールは必ず Gmail 下書きに留め、人間が内容（特に単金・商流の記載）を確認してから送信する。単金を先方に開示するかは商習慣上センシティブなため、下書きテンプレートでは要員側への提示単金を空欄またはプレースホルダにするか選択できるようにする
2. **誤抽出の想定**: 単金の表記ゆれ（万円/月、円/時、「〜80」）は月額万円に正規化するが、換算ミスがあり得るため、マッチDBには抽出元メールへの参照を必ず残し、ワンクリックで原文確認できるようにする
3. **粗利下限の変更**: `.env` の `MIN_GROSS_MARGIN_JPY` を書き換えるだけで次回バッチから反映。市況に応じて調整する
4. **個人情報の取り扱い**: 要員情報（氏名・年齢・居住地）は PII。ZDR キーの利用、Notion ワークスペースのアクセス制限、不要になった要員レコードの定期削除ルールを運用に含める
5. **転送設定の監視**: A案（Gmail転送）は Xserver 側の転送設定が生命線。1日2回のバッチで受信0件が続いた場合はサマリメールで警告する
6. **成約・見送りのフィードバック**: マッチDBのステータスを更新する運用を続けると、将来的に「どんなマッチが成約しやすいか」を判定プロンプトに反映できる（Phase 3 以降の改善余地）
