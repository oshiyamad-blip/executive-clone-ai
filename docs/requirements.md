# 要件定義書たたき台：経営者クローンAIシステム

## 1. プロジェクトの背景と目的

- 経営者の意思決定の仕組みや思考をデジタル化して再現する
- 経営企画や部門長が経営者と会議を行う前にAIと壁打ちを行う
- この事前準備により会議の回数を削減して意思決定の速度を劇的に高める
- 経営者自身が気づいていない潜在的な行動の因果関係を抽出して経営の資産とする

## 2. システム全体像

- 本システムは経営者の日常的な言動をすべて集約する
- その中から経営に重要な情報のみをシグナルとして取り出す
- 過去の意思決定の文脈をストーリーとして整理する
- これらを統合したデータベースを基盤に経営者の思考を模倣する対話インターフェースを提供する

## 3. 機能要件

### 3.1. マルチソースデータ収集機能

経営者の言動データをあらゆるチャネルから自動で収集して集約する。

連携対象データ:

| ソース | 内容 |
|--------|------|
| ビジネスチャット | SlackやChatworkの投稿データ |
| 電子メール | 経営者が送受信したビジネスメール |
| カレンダー | Googleカレンダーの予定履歴と予定詳細 |
| オンライン会議 | Google Meetなどの録音データと自動文字起こしテキスト |
| メッセンジャー | LINEなどの対外的なやり取り |
| ライフログ | 身につけている録音デバイスから取得する対面会議や独り言の日常音声データ |
| 各種文書 | Googleドキュメントなどの作成資料 |

### 3.2. シグナル抽出機能

収集した膨大なテキスト情報から無駄な雑談や挨拶を除去する。
経営や事業に影響を与える重要情報のみを抽出してシグナルデータベースに格納する。

抽出対象の基準:

| カテゴリ | 内容 |
|----------|------|
| hypothesis | 新しい仮説や気づき |
| key_person | 外部の人材や重要人物との接触 |
| idea | 新規事業や既存事業に関するアイデア |
| decision | 経営方針や重要な意思決定 |
| trend | 業界トレンドや競合他社に関する知識 |

### 3.3. ストーリー分析・構築機能

断片的な情報を時系列で繋ぎ合わせて因果関係を持つストーリーとして整理し、
ストーリーデータベースに蓄積する。

構築ステップ:

1. **初期設定**: 経営者自身の過去の成功体験や失敗体験をパターンとして数十個あらかじめ手動で登録する
2. **自動抽出**: シグナルデータベースを週次・月次・年次の単位で時系列分析する
3. **関連性の検出**: ある人物との出会いやアドバイスがその後の事業進捗や業績向上にどう繋がったかを自動検出して新しいストーリーとして登録する

### 3.4. 意思決定シミュレーション対話機能

ユーザーが事業方針や課題を相談した際に経営者の分身として回答を出力する。

想定ユーザー: 経営企画・役員・各事業部長

機能詳細:

| 機能 | 内容 |
|------|------|
| 経営理念プロンプト | 経営者が掲げる優先事項や15個程度の意思決定ルールを基盤として設定 |
| 思考の再現 | 質問に対して経営者ならどのように判断するかを模倣した回答を提示 |
| 根拠の明示 | 回答の根拠となったシグナル・ストーリーDBの参照元を合わせて提示 |
| 疑似ログ再入力 | AIとユーザーがやり取りしたテキストを再びDBに食わせて学習ソースとして循環 |

### 3.5. データの名寄せと統合機能

オンライン会議の自動文字起こしテキストと音声ライフログなどの重複データを検知して
1つのログにマージする。

## 4. 非機能要件

### 4.1. セキュリティと権限管理

経営の極秘情報を扱うため厳格な権限設計を行う。

| 要件 | 内容 |
|------|------|
| アクセス権限 | 経営幹部や特定の承認者のみが対話インターフェースを使用できる |
| 参照制限 | データベースごとにアクセス可能なユーザー層を分離する |
| データ保護 | 外部LLMモデルにデータが送信されても二次利用・学習に利用されないセキュアなAPI環境を使用する |

### 4.2. システム反映頻度

| 処理 | 頻度 |
|------|------|
| データ同期 | 日常データは1日1回バッチ処理でシグナルデータベースへ反映 |
| ストーリー生成 | 週末などの低負荷時間帯に週次でストーリー抽出処理を実行 |

## 5. 開発プロセスと想定技術

| 項目 | 内容 |
|------|------|
| 開発手法 | Claude Codeを活用し機動的にプロトタイプを作成しながら週単位で進化させる |
| 連携ツール | セキュリティの壁を越えるためにMCPなどを構築して安全にGoogleドキュメントなどの社内データを取得する |
| データベース | 一覧性と人間による目視確認を容易にするためNotionなどをデータベースの受け皿として活用する |

## 6. 導入・運用上の注意点

- **社内情報のデジタル化が前提**: アナログな情報や手書きのメモが多い場合はデータ収集の段階で停滞する
- **経営者の全面的な協力**: 日々の録音やログ収集に経営者本人が心理的に順応して自らAIと対話を繰り返すコミットメントが不可欠
